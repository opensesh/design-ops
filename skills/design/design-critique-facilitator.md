# Design Critique Facilitator

Run structured design critiques with clear roles, frameworks, and actionable outcomes.

## Purpose

Facilitate productive design critiques that surface useful feedback, avoid subjective debates, and result in clear next steps. Can be used for self-critique or facilitating team discussions.

## When to Activate

Use this skill when:
- Reviewing designs before handoff
- Running team critique sessions
- Self-critiquing your own work
- Preparing designs for stakeholder presentation

---

## Critique Framework

### The Core Question

Every critique point should answer: **"Does this design effectively solve the stated problem for the intended user?"**

Not: "Do I like this?"

### Critique Dimensions

Evaluate designs across these dimensions:

1. **Problem Alignment** - Does it solve the right problem?
2. **User Needs** - Does it serve user goals?
3. **Hierarchy & Flow** - Is the information structured well?
4. **Visual Design** - Is it aesthetically appropriate?
5. **Usability** - Can users accomplish tasks?
6. **Consistency** - Does it fit the system?
7. **Edge Cases** - What happens at extremes?

---

## Critique Process

### Step 1: Context Setting (2 min)

Before critique, establish:
- **The problem:** What is this design solving?
- **The user:** Who is this for?
- **The constraints:** What limitations exist?
- **The ask:** What feedback is most useful?

```
"This is a [feature/page] for [user type] who needs to [goal].
We're constrained by [constraints].
I'd especially like feedback on [specific area]."
```

### Step 2: Observation (5 min)

Describe what you see before judging:
- What's the visual hierarchy?
- What's the flow?
- What elements are present?
- What's emphasized, what's de-emphasized?

### Step 3: Analysis (10 min)

Evaluate against the dimensions:

```
## Design Critique: [Design Name]

### Problem Alignment
- [ ] Clear what problem this solves
- [ ] Solution matches problem scope
- [ ] Success criteria are evident

**Notes:** [Observations]

### User Needs
- [ ] Primary user goal is supported
- [ ] User journey is logical
- [ ] User context is considered

**Notes:** [Observations]

### Hierarchy & Flow
- [ ] Most important elements are prominent
- [ ] Reading order is logical
- [ ] Grouping makes sense

**Notes:** [Observations]

### Visual Design
- [ ] Aesthetically appropriate for context
- [ ] Consistent with brand
- [ ] Typography is clear and readable
- [ ] Color use is purposeful

**Notes:** [Observations]

### Usability
- [ ] Interactive elements are obvious
- [ ] Actions are clearly labeled
- [ ] Feedback states are considered
- [ ] Error handling is addressed

**Notes:** [Observations]

### Consistency
- [ ] Follows design system
- [ ] Patterns match existing product
- [ ] Terminology is consistent

**Notes:** [Observations]

### Edge Cases
- [ ] Empty states
- [ ] Maximum content
- [ ] Minimum content
- [ ] Error states
- [ ] Loading states

**Notes:** [Observations]
```

### Step 4: Feedback Delivery (10 min)

Structure feedback clearly:

```
## Feedback Summary

### Strengths
- [What's working well and why]
- [Specific example]

### Questions
- [Clarifying question]
- [Assumption to validate]

### Concerns
- [Issue] - [Why it matters] - [Possible solution]
- [Issue] - [Why it matters] - [Possible solution]

### Suggestions
- [Idea to explore]
- [Alternative approach]
```

### Step 5: Next Steps (3 min)

Close with clear actions:
- What changes are being made?
- What needs further exploration?
- When is the next review?

---

## Feedback Formulas

**For concerns:**
"I notice [observation]. I'm concerned because [reason]. Have you considered [suggestion]?"

**For questions:**
"Help me understand [element]. What's the thinking behind [decision]?"

**For praise:**
"[Element] works well because [reason]. It helps achieve [goal]."

**Avoid:**
- "I don't like..." (subjective)
- "You should..." (prescriptive)
- "This is wrong" (unconstructive)

---

## Self-Critique Checklist

When critiquing your own work:

**Before presenting:**
- [ ] Can I articulate the problem this solves?
- [ ] Have I considered the primary user journey?
- [ ] Does the hierarchy match importance?
- [ ] Have I checked against the design system?
- [ ] Did I explore alternatives?
- [ ] What are the weakest points?
- [ ] What feedback do I need?

**Common blind spots:**
- Edge cases (empty, error, loading)
- Mobile/responsive behavior
- Accessibility (contrast, touch targets)
- Real content vs placeholder text
- First-time vs returning user experience

## Critique Session Facilitation

For team critiques:

### Roles

**Presenter:** Shares context, listens, doesn't defend
**Facilitator:** Keeps discussion on track, ensures equity
**Participants:** Ask questions, provide feedback

### Ground Rules

1. Critique the work, not the person
2. Questions before judgments
3. Be specific, not vague
4. Suggest, don't prescribe
5. Stay focused on the stated problem

### Time Structure (30 min session)

| Phase | Time | Activity |
|-------|------|----------|
| Setup | 2 min | Context from presenter |
| Silent review | 3 min | Everyone observes quietly |
| Questions | 5 min | Clarifying questions only |
| Feedback | 15 min | Observations and suggestions |
| Synthesis | 5 min | Key takeaways and next steps |
